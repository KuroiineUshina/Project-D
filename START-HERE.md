# Project D 새 세션 시작 지침

이 문서는 다른 컴퓨터, Codex 서브 에이전트, 다른 세션에서 Project D 작업을 이어갈 때 가장 먼저 읽는 진입점이다.

## 현재 목표

Project D의 현재 목표는 전체 게임을 한 번에 만드는 것이 아니라, 먼저 20분~30분짜리 첫 수직 슬라이스를 완성하는 것이다.

핵심 루프:

1. 오두막에서 탐사 준비를 한다.
2. 작은 숲 구역으로 탐사를 나간다.
3. 오염 흔적을 조사하고 아이템을 수색한다.
4. 약한 위협을 피하거나 처리한다.
5. 살아서 오두막으로 귀환한다.
6. 전리품을 분류하고 제작/의뢰/거점 상태가 갱신된다.
7. 다음 탐사 목표가 생긴다.

## 반드시 지킬 규칙

- 이미지는 무조건 imagegen 스킬로 생성한다.
- UI는 Unity Canvas 우선으로 구현한다.
- 코드나 벡터로 이미지 자체를 생성하지 않는다.
- 저장은 단일 저장 파일만 사용한다. `새 게임`/`이어하기`는 분리하지 않는다.
- 진행 초기화는 `환경설정 -> 기타 -> 세계 초기화`에서 처리한다.
- 탐사에는 맵별 제한시간이 있다. 초기 개발 단계에서 움브라데 숲은 25분 제한이다.
- 타임오버나 치명상은 사망/실종이 아니라, 디아나가 짐을 내던지고 오두막으로 급하게 귀환하는 실패로 처리한다.
- 스토리, 의뢰 라인, 지역 설정, NPC 설정은 사용자가 직접 작성한다. Codex는 질문과 피드백을 담당한다.
- 기획, 보고서, 작업 요약은 기본적으로 한국어로 작성한다.
- 새 시스템을 추가하기 전에 첫 수직 슬라이스에 필요한지 확인한다.
- 큰 구현보다 먼저 작게 플레이 가능한 루프를 만든다.

## 작업 시작 체크리스트

1. 최신 원격 상태를 받는다.

```powershell
git pull
```

2. 아래 문서를 순서대로 읽는다.

- [AGENTS.md](AGENTS.md)
- [프로젝트 방향과 레퍼런스](docs/project-direction.md)
- [첫 플레이 가능 수직 슬라이스](docs/design/first-playable-slice.md)
- [분야 분담과 책임 범위](docs/process/workstreams.md)
- [협업 및 Git 작업 흐름](docs/process/collaboration-workflow.md)
- [프로젝트 결정표](docs/design/project-decisions.md)
- [작업 기록 보관소](docs/logs/SESSION-LOG.md)
- [Escape from Tarkov 조사 인덱스](docs/research/escape-from-tarkov-research-index.md)

3. 현재 작업이 어느 분야에 속하는지 정한다.

- 기획/디렉션
- 서사/세계관/의뢰
- 플레이어/카메라/조작
- 탐사/귀환/정리
- 아이템/인벤토리/수색
- 제작/거점/오두막
- 전투/적/생존
- 맵/레벨/환경 연출
- UI/UX
- 기술 기반/저장/Git

4. 작업 전 상태를 확인한다.

```powershell
git status --short --branch
```

5. 실험성 작업은 별도 브랜치에서 진행한다.

```powershell
git checkout -b design/topic-name
```

## 현재 확정된 초반 흐름

1. 메인 메뉴와 환경설정
2. 짧고 조작 가능한 마법 폭주 프롤로그
3. 디아나가 마력을 잃고 움브라데 숲으로 밀려남
4. 낡은 오두막을 최소한으로 정비하며 채집/제작/수리를 배움
5. 같은 숲의 분위기가 안개, 사운드, 오염 흔적으로 변함
6. 오염된 시체를 발견하고 오두막으로 돌아옴
7. 탐사 준비 화면에서 무기, 회복품, 가방, 목표를 확인함
8. 첫 숲 탐사에서 약한 감염체와 오염 흔적을 확인함
9. 살아서 귀환해 전리품을 분류함
10. 첫 의뢰가 열리며 본 게임 루프가 시작됨

## 현재 피해야 할 범위

- 자유 건축 전체 시스템
- 복잡한 동물 생태계
- 인간형 무법자 전투 AI
- 고품질 시네마틱 다수
- 대형 오픈월드 숲
- 타르코프식 깊은 장비/부상/경제 전체 구현
- 복잡한 상인 경제와 거점 확장 트리

지금은 이 모든 것보다 `준비 -> 탐사 -> 귀환 -> 정리 -> 제작`이 재미있는지 먼저 증명한다.

## 세션 종료 전 체크리스트

작업을 마치기 전 다음을 확인한다.

- 관련 문서를 업데이트했는가?
- 변경 범위가 첫 수직 슬라이스 목표와 맞는가?
- `git status`가 의도한 변경만 보여주는가?
- 가능하면 검증 명령이나 실행 결과를 남겼는가?
- [작업 기록 보관소](docs/logs/SESSION-LOG.md)에 작업 내용을 남겼는가?
- 커밋 메시지가 작업 내용을 설명하는가?
- GitHub에 push했는가?
